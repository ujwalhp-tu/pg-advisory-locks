# Aura Rules for pg-advisory-lock

## Architecture & Conventions
- **Framework**: Spring Boot 3.2.5, Java 21, Gradle.
- **Distributed Locking**: Uses PostgreSQL transaction-level advisory locks (`pg_advisory_xact_lock` and `pg_try_advisory_xact_lock`).
- **Lock Namespaces**: All lock namespaces must be defined in the centralized `LockResource` enum to survive refactoring and blue-green deployments. Do not use raw strings or reflection-based names (class/method names).
- **Lock Hashing**: Lock keys are generated by concatenating the namespace and resource ID, then hashing them into a 64-bit integer using Guava's `Hashing.murmur3_128().asLong()`. This provides superior collision resistance over Postgres's 32-bit `hashtext()`.
- **Transaction Scope**: The lock service uses `@Transactional(propagation = Propagation.REQUIRED)`. Locks will join the caller's existing transaction and will only be released when the *outer* transaction commits or rolls back.
- **Global Locks**: For processes without a specific entity ID (e.g., cron jobs), use the overloaded methods that omit the `resourceId` (defaults to `"GLOBAL"`).
- **Timeouts**: Prefer methods with a `Duration timeout` to prevent indefinite blocking, unless strictly required by the business logic.
